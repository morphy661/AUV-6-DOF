# 六自由度混合推进器诊断阶段结果

更新日期：2026-07-15

## 本阶段完成内容

- 为 H1–H4、V1–V2 增加独立的 RPM、电压和电机温度遥测。
- `NO_OUTPUT` 同时降低实际推力、电流和 RPM；`THRUST_LOSS` 只降低实际推力，保持电流和 RPM 近似正常，用于模拟叶片损伤、空化等水动力故障。
- 为每台推进器计算电流比、RPM 比和本地异常分数。
- 使用 IMU 线加速度、角速度变化、推进器几何和指令力矩，把整车响应残差投影到六台推进器。
- 实际推力、推力效率和故障标签仍只用于标签与离线评估，不进入模型输入。
- 原始特征由 61 维扩展到 109 维，加入一阶时间差后模型输入为 218 维。
- 建立双路径模型：旧的整车运动特征进入主 BiLSTM；新增推进器遥测按 H1–V2 重排后进入独立局部时序分支；两个分支在模式头和位置头之前融合。

## 数据与验证

- 260 个独立任务，13 个联合场景，每个场景 20 个任务。
- 18,639 个五秒因果窗口，序列形状为 `100 × 109`。
- 固定任务划分：训练 182、验证 39、域外压力测试 39。
- 完整数据文件为 `simulation_dataset_six_dof_hybrid_telemetry.pth`，大小 813,684,496 字节。
- SHA-256：`8A2E7D7B053469BC82B7A9501BBE711C2ED7BF577F7D61AB4C964D15CA0FE318`。
- 生成数据和模型结果被 `.gitignore` 排除，不提交到 GitHub。

## 当前结论与下一步

本阶段证明了“本地推进器检查 + 整车异常确认”是正确方向，但还不能直接驱动 FTC。下一步应保持模型主体不变，优先处理：

1. 在验证任务上设计置信度阈值、连续窗口确认、进入/退出滞回和最短保持时间，降低 28.48% 的正常误报，同时报告增加的检测延迟。
2. 把当前对角质量矩阵的瞬时响应投影升级为完整力观测残差：`M*nu_dot + C(nu)*nu + D(nu)*nu + g - tau_command`，并使用故障前数据校准参数。
3. 对六类推力损失单独报告召回率、定位 F1 和混淆矩阵，重点检查 H3 以及轻微推力损失。
4. 方案定型后使用新的随机种子命名空间生成从未查看过的最终测试集，只评估一次。

在误报率和推力损失定位达到门槛前，诊断输出只应以影子模式记录，不应自动改变控制分配。

## 多时间尺度故障确认层（原精确定位目标）

根据真实海下运行中“短时运动异常可能被闭环控制自行修正”的情况，新增独立于神经网络的因果状态机：

`normal → suspected → confirmed → recovering → normal`

- 无输出和推力损失使用不同的确认时间。
- 推力损失只有在多个窗口持续出现后才进入确认状态。
- 已确认故障只有在正常概率持续满足恢复条件后才清除。
- 推进器位置变化使用独立滞回，避免 H1–V2 在相邻窗口间跳动。
- 每个任务维护独立状态，所有阈值只使用验证任务校准。

验证集选择的参数为：

| 参数 | 数值 |
|---|---:|
| 故障进入概率 | 0.55 |
| 无输出确认时间 | 0 s |
| 推力损失确认时间 | 2.5 s |
| 正常退出概率 | 0.75 |
| 恢复确认时间 | 3.75 s |
| 位置确认时间 | 1.25 s |

无输出由电流与 RPM 的直接异常支持，因此允许快速确认；仅依靠运动响应的推力损失必须持续 2.5 秒。最终配置从 384 组模式参数和 20 组位置参数中选择，域外任务没有参与选择。

这项结果支持把时序层用于故障存在性确认，但暂不应用它直接触发推进器隔离或 FTC 重构。该配置保留为历史对照；下面的维护型目标取代“必须精确命中 H1–V2”成为当前主评价标准。

## 维护型健康监测目标

考虑到轻度推力损失可能被闭环控制补偿，当前系统不再把六台推进器精确定位作为部署合格条件。诊断输出改为四级：

| 等级 | 含义 | 默认动作 |
|---|---|---|
| 正常 | 没有持续异常 | 不记录 |
| 短时观察 | 异常尚未持续到确认时间 | 内部记录并继续观察，不触发 FTC |
| 持续退化 | 推力损失持续存在但控制仍可补偿 | 任务后检查，不自动改变控制分配 |
| 严重故障 | 无输出，或跟踪误差/控制饱和达到安全阈值 | 安全告警并考虑 FTC |

位置分支只输出 `水平/垂直/不确定` 组别、Top-2 推进器候选及置信度，供人工检查排序。精确 Top-1 和 13 类联合结果只保留为研究对照。

维护输出进一步拆成两层：

- **原始健康日志**保留时序异常、模式和候选推进器，用于回放与研究，目标是尽量不漏掉故障。
- **正式维护工单**才会提示人工检查。无输出在足够激励和本地电流/RPM异常一致时走快速通道；推力损失除足够激励和独立运动损失证据外，还要在同一个稳定目标/航段上下文内累计 8 s 合格证据。跨目标切换的待确认状态会被拆分，跨越上下文边界的窗口不参与开单确认，恢复后的复发只形成间歇性提示而不自动升级为工单。若连续恢复 3.75 s，则关闭待确认状态但保留观察记录。低激励或证据不足的异常仍留在日志中，但不形成工单。

时序层从 384 组候选中只用验证任务选择，当前配置为：

| 参数 | 数值 |
|---|---:|
| 故障进入概率 | 0.55 |
| 无输出确认时间 | 0 s |
| 推力损失确认时间 | 2.5 s |
| 正常退出概率 | 0.75 |
| 恢复确认时间 | 3.75 s |

正式工单层在 V2 验证任务上所选参数为：最小激励比例 0.05、最小运动损失证据 0.10、最小本地无输出异常 0.50、无输出工单确认 0 s、推力损失同上下文累计确认 8 s、待确认恢复取消 3.75 s、活动工单恢复确认 1.25 s、复发观察窗口和同模式工单合并间隔均为 30 s。目标/航段编号只作为决策层上下文边界，不进入 BiLSTM-Attention 的输入特征；所有故障证据只使用当前及历史可观测窗口，不读取故障注入真值。

两级设计把“是否值得记录”和“是否值得打扰维护人员”分开：原始日志继续覆盖 36/36 个故障任务；域外测试中的正式误工单由原先 4 个进一步降到 0，无输出仍保持 18/18 开单。代价是正式推力损失工单只覆盖 2/18；其余异常没有丢失，而是进入原始日志或待确认观察记录。本次测试共产生 33 条待确认观察，其中 31 条在恢复后取消、2 条被后续无输出快速通道取代。该配置明确偏向“零误维护”：适合维护辅助和任务后检查，不适合把正式工单召回率解释成完整故障检测召回率。

因此当前版本适合作为维护辅助和影子健康日志，而不是让模型位置分类直接隔离推进器。正式工单只影响维护提示；FTC 的定向重分配仍要求安全监督器自己的直接 ESC 无输出证据。结果也不代表最终无偏泛化能力：这 39 个域外任务已经参与多轮方案比较，设计冻结后必须使用新随机种子生成一次性最终测试集。主训练数据仍没有把“短时异常后恢复”作为故障类别；下面使用完全独立、全程无故障标签的瞬态基准进行验证，避免把该测试集重新用于训练。

## 短时异常后恢复基准

现在已增加不注入推进器故障的专用瞬态基准，覆盖纵向海流、横向/偏航海流、垂向海流和多轴湍流四种半正弦外力脉冲。每种场景使用 3 个独立噪声种子，共 12 个任务。脉冲持续 1.00–1.75 s，均短于 FTC 的 2 s 控制压力确认时间；仿真继续运行到 45 s，以检查控制器是否真正恢复，而不是只检查扰动结束瞬间。

合格条件为：必须观察到运动偏差并恢复；允许写入原始健康日志，但不得形成正式维护工单，也不得发生 FTC 推力重分配、推进器隔离、任务终止或应急上浮请求。

| 指标 | 结果 |
|---|---:|
| 瞬态任务 | 12 |
| 观察到运动异常并恢复 | **12/12** |
| 峰值位置误差范围 | 0.202–0.382 m |
| 最大最终位置误差 | 0.040 m |
| 最大峰值姿态误差 | 0.201 rad |
| 最大最终姿态误差 | 0.0031 rad |
| 原始健康日志事件 | 4 |
| 正式维护工单 | **0** |
| FTC 干预任务 | **0** |

其中 3 个多轴湍流任务被模型暂时记录为疑似推力损失，另有 1 个纵向海流任务形成短时观察记录；它们在车辆恢复后都没有通过“足够激励 + 独立故障证据”的正式工单门控。这说明当前两级设计能够保留可供研究的异常线索，同时避免把已经自行恢复的外部扰动直接转化为维护要求。

该结果仍是第一版仿真基准，而不是真实海试认证。后续应扩展脉冲持续时间和强度边界，并加入推进器附近流场遮挡、缆绳瞬时拖曳、DVL 短时失锁和波浪诱导姿态扰动。

## 安全边界扫描

在第一版瞬态基准之后，使用最容易触发推力损失误判的多轴湍流进行紧凑边界扫描：持续时间为 1、2、4 s，强度为弱、中、强；另加入 1、2、4 s 的 DVL 完全失锁。每个设置使用 3 个独立噪声种子，共 36 个全程无推进器故障任务。

| 场景 | 1 s | 2 s | 4 s |
|---|---:|---:|---:|
| 弱多轴扰动 | 3/3 无干预安全恢复 | 3/3 无干预安全恢复 | 3/3 无干预安全恢复 |
| 中等多轴扰动 | 3/3 无干预安全恢复 | 3/3 FTC 降级观察后恢复 | 3/3 FTC 降级观察后恢复 |
| 强多轴扰动 | 3/3 无干预安全恢复 | 3/3 FTC 降级观察后恢复 | 3/3 FTC 降级观察后恢复；正式维护工单 0 |
| DVL 完全失锁 | 3/3 无干预安全恢复 | 3/3 无干预安全恢复 | 3/3 无干预安全恢复 |

全部 36 个任务最终都恢复，且修改后没有任何 `SAFE_HOLD_OR_ABORT`、`CONTROLLED_ASCENT` 或推进器隔离动作。离散扫描得到两个不同边界：

- 弱多轴扰动至少可持续到已测试的 4 s。
- 中等扰动的“完全无动作”边界为 1 s，但允许观察性降级时安全边界至少达到 4 s。
- 强扰动的“完全无动作”边界为 1 s；允许观察性降级且不生成维护工单时，安全边界至少达到已测试的 4 s。
- DVL 完全失锁至少可持续到已测试的 4 s，而不触发维护工单或 FTC。

FTC 已据此改为双通道临界确认：推进器故障证据与临界控制压力同时存在时仍使用 2 s 快速安全升级；只有跟踪/分配压力而没有推进器证据时，先在 2 s 进入 `DEGRADED_OPERATION`，持续 5 s 且没有至少 20% 的恢复趋势才允许 `SAFE_HOLD_OR_ABORT`。直接 ESC 电流/RPM 无输出仍在 0.5 s 确认并优先执行一次定向重分配。相同 36 个任务修改前有 11 个出现降级或终止，其中 8 个出现 `SAFE_HOLD_OR_ABORT`；修改后 12 个只进入观察性降级，危险终止/上浮/隔离动作从 8 个降为 0。

新增的待确认工单状态机解决了强 4 s 外部扰动误维护问题：推力损失需累计确认 8 s，恢复 3.75 s 后取消正式升级并保留观察记录，30 s 内复发只累加可疑时长。相同 36 个无故障任务中，正式工单、危险终止、上浮和推进器隔离均为 0；12 个中强度 2–4 s 任务仍进入观察性降级但全部恢复，另外 2 条单窗口推力损失证据在恢复后只作为待确认观察保留。

这里的边界只位于离散的测试点之间，不等于精确连续阈值。若要确定最终参数，应在 1–2 s 区间再测试少量 1.5 s 和 1.75 s 点，而不需要重新扩展成大规模数据集。

## V2 上下文感知修复与新种子盲测

V1 失败后没有修改或重跑原盲测，而是实现了独立的 V2 状态机：控制目标携带非负 `guidance_context_id`，窗口构建器另外输出上下文编号及“整个因果窗口是否处于同一上下文”的稳定标志。它们只供维护决策层切分证据，不进入 109 维模型输入，因此不是故障标签或故障位置泄漏。

V2 正式开单逻辑有四个关键约束：

- 推力损失的 8 s 确认时间只能由同一稳定上下文内、实际连续合格的证据累计；无证据间隔不计时。
- 目标/航段切换会把尚未确认的推力损失保存为 `context_transition_observation`，并清除跨工况复发历史；上下文不稳定窗口仍进入原始日志，但不能促成正式工单。
- 已恢复后再次出现的推力损失可生成 `intermittent_thrust_loss_advisory`，不再因为“出现两次”直接形成正式工单。
- 已经成立的正式工单不会因目标切换被隐藏；无输出的直接 ESC 快速通道也保持不变。

## 事后日志合并与分级

V2 盲测归档后增加了独立的日志呈现层 `maintenance_log_policy.py`。它不修改模型概率、健康等级、正式工单数组或 FTC 输出，只对已经保留的原始事件进行组织：

- 连续异常始终保持一个事件；两个已经分开的片段只有在同一任务、同一故障模式、同一稳定目标上下文且间隔不超过 5 s 时才合并。
- `background_trace`：短时、未确认的异常，仅归档。
- `observation`：持续存在但没有形成正式工单的异常，在折叠观察区保留。
- `maintenance_advisory`：已经形成正式推力损失工单，进入任务后维护检查队列。
- `safety_alert`：已经形成正式无输出工单，要求立即关注；具体隔离仍由独立 FTC 证据决定。

## 六自由度 FTC 安全监督器

新FTC不允许模型类别直接控制推进器。信息流固定为：

`模型/规则证据 → 健康等级 → FTC安全监督器 → 推力分配器或任务安全请求`

更新后的动作和触发条件如下：

| 动作 | 触发条件 |
|---|---|
| `NORMAL_CONTROL` | 无持续故障和控制压力 |
| `LOG_ONLY` | 短时异常或可补偿推力损失 |
| `DEGRADED_OPERATION` | 中等跟踪/分配压力持续 2 s；或故障证据伴随持续控制饱和，用于降低需求并观察恢复 |
| `TARGETED_REALLOCATION` | 某台推进器在足够激励下，电流与RPM无输出分数具有明确优势并持续0.5秒 |
| `SAFE_HOLD_OR_ABORT` | 临界压力与推进器故障证据同时持续 2 s；或无推进器证据的临界压力持续 5 s 且未出现至少 20% 恢复趋势 |
| `CONTROLLED_ASCENT` | 垂直控制能力不可用；向独立安全上浮机构发出请求 |

安全约束：

- 只有ESC电流和RPM的独立无输出证据能够关闭具体推进器；Top-1或Top-2模型输出本身不能关闭推进器。
- 激励不足时不允许定向隔离，即使模型判断无输出也只记录。
- 普通高控制量在没有故障证据时不会单独触发降级。
- 已确认无输出的推进器在本次任务内保持隔离，防止反复接通。
- 推力损失在控制器能够补偿时保持`LOG_ONLY`；只有独立控制压力持续存在才升级。
- 规则证据构造器明确不读取`actual_thruster_forces`、`thruster_force_efficiencies`、`faulted_thruster_index`或故障注入标签。

六自由度仿真器已经支持可选FTC监督器。监督器给出的估计效率在下一控制步进入有界推力分配，替代Oracle真值。真实H1无输出闭环测试中，仅凭可观测ESC证据在0.6秒确认H1，并在下一步将H1估计效率置零；相同接口对可补偿推力损失只记录、不隔离。

当前`CONTROLLED_ASCENT`只是向任务管理器发出的安全请求，尚未模拟抛载、浮力释放或压载舱动作，不能把缩放推进器指令等同于真实应急上浮。这是后续安全执行层需要补充的内容。
